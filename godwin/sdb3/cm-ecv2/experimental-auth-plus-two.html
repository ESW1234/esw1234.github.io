Experimental design:
  • showChatButton / hideChatButton short-circuit in CM deployments — emit onEmbeddedMessagingChannelMenuVisibilityChanged only, never touch the iframe FAB.
  • toggleChatFabVisibility drops its CM branch (now reachable only via launchChat post-handoff or non-CM).
  • businessHoursTimerCallback just calls utilAPI.show/hideChatButton — the short-circuit handles CM/non-CM dispatch internally.
  • Lifecycle paths (setIdentityToken, resetClientToInitialState, handleFabReadyEvent, handleMinimize) keep their cwcOwnsFab reads — these aren't customer-facing API entry points.
Compare against the canonical auth version at auth-plus-two.html, or the experimental unauth at experimental.html.
Auth-mode scenarios to test:
  1. Page load → CM emits {isVisible:false} (no JWT yet, AUTH gate fails). MIAW item should be hidden in CM dropdown.
  2. Click Get JWT & Set → setIdentityToken fires → CM emits {isVisible:true} → MIAW reorders into the dropdown.
  3. Click MIAW in the dropdown → modal opens. (cwcOwnsFab=true, latch flipped.)
  4. From console: embeddedservice_bootstrap.utilAPI.hideChatButton() → experimental: CM emits :false only, modal stays open. Old behavior post-handoff: would also try to hide the iframe FAB.
  5. Click Clear session → resetClientToInitialState → identityToken cleared → CM emits {isVisible:false} via the lifecycle path.
  6. Re-auth: click Get JWT & Set again → setIdentityToken hits the cwcOwnsFab && cwcFabReadyHasFired branch (still present in lifecycle paths) → unhide iframe directly.
